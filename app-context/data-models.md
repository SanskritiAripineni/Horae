# AutoLife — Data Models

## Journal Entry (mobile → backend)

Journals are created by the background Android service (or manually) and sent to the backend.

```json
{
  "id": "entry_123",
  "entry_number": 1,
  "created_at": "2024-01-15 14:30",
  "period": "Morning",
  "content": "Woke up tired. Had coffee and went for a 30-minute run. Felt energized afterward.",
  "timestamp": "2024-01-15 14:30"
}
```

| Field | Type | Notes |
|-------|------|-------|
| `id` | string | Local database row ID |
| `entry_number` | int | Sequential index |
| `created_at` | string | Human-readable datetime |
| `period` | string | Morning / Afternoon / Evening / Night |
| `content` | string | Journal text (Gemini-generated or manual) |
| `timestamp` | string | Same as `created_at` |

## Raw Day Markers (sensor aggregates)

One dict per calendar day. Generated by Android sensors, sent alongside journals.

```json
{
  "date": "2024-01-15",
  "sleep_onset_hour": 23.5,
  "sleep_duration_hours": 6.5,
  "sleep_regularity_index": 72.0,
  "late_night_screen_min": 60.0,
  "total_screen_min": 300.0,
  "app_switching_rate": 2.8,
  "mobility_entropy": 0.70,
  "location_revisit_ratio": 0.68,
  "social_rhythm_metric": 0.75,
  "comm_reciprocity": 0.50,
  "coverage": {
    "sleep_onset_hour": 1.0,
    "sleep_duration_hours": 1.0,
    "app_switching_rate": 0.9
  }
}
```

| Field | Range | Meaning |
|-------|-------|---------|
| `sleep_onset_hour` | 0–24 | Hour of night when sleep began |
| `sleep_duration_hours` | 0–12 | Total sleep hours |
| `sleep_regularity_index` | 0–100 | Consistency of sleep/wake times (higher = more regular) |
| `late_night_screen_min` | 0–360 | Screen time after 10 PM |
| `total_screen_min` | 0–720 | Total daily screen time |
| `app_switching_rate` | 0–10 | Average app switches per minute (higher = more distracted) |
| `mobility_entropy` | 0–1 | Diversity of locations visited (higher = more varied movement) |
| `location_revisit_ratio` | 0–1 | Fraction of time in familiar locations |
| `social_rhythm_metric` | 0–1 | Regularity of social interactions |
| `comm_reciprocity` | 0–1 | Balance of sent vs. received communications |
| `coverage` | per-field 0–1 | Data confidence/completeness for each marker |

## Proposed Calendar Change

```json
{
  "action": "add",
  "title": "Evening Wind-Down",
  "description": "Phone-free buffer to prepare for sleep",
  "start_time": "2024-01-16T21:00:00",
  "end_time": "2024-01-16T21:30:00",
  "category": "Sleep",
  "reason": "Address late-night screen use by creating a boundary before bed",
  "duration_hours": 0.5
}
```

| Field | Values | Notes |
|-------|--------|-------|
| `action` | `add` / `update` / `delete` | Calendar operation |
| `title` | string | Event title |
| `description` | string | Brief description |
| `start_time` | ISO 8601 | `YYYY-MM-DDTHH:MM:SS` |
| `end_time` | ISO 8601 | `YYYY-MM-DDTHH:MM:SS` |
| `category` | Sleep / Stress / Social / Physical / Mindfulness / Wellness / Task | Wellness domain |
| `reason` | string | Why the AI suggests this |
| `duration_hours` | float | Optional pre-computed duration |

## Wellbeing Assessment (pipeline output)

```json
{
  "risk_level": "mild",
  "key_concerns": [
    "below-average sleep duration",
    "elevated late-night screen time"
  ],
  "positive_indicators": [
    "maintained physical activity",
    "social engagement present"
  ],
  "behavioral_context": "Recent sleep disruption likely due to screen use; exercise routine remains strong."
}
```

Risk levels: `minimal` → `mild` → `moderate` → `severe`

## Behavioral Sensing Output

```json
{
  "prose": "Sleep disruption detected with elevated late-night device engagement. Physical activity maintained despite reduced rest.",
  "behavioral_state": {
    "concerns": ["disrupted sleep onset", "elevated screen time"],
    "positives": ["consistent physical activity"],
    "categories_affected": ["sleep", "screen"],
    "severity": "moderate"
  },
  "llm_analysis": {
    "coherent_patterns": ["screen-sleep disruption cycle"],
    "suggestions": [
      {
        "start_time": "21:30",
        "end_time": "21:45",
        "change": "Screen-free wind-down ritual",
        "rationale": "Break late-night habit, restore sleep boundary"
      }
    ]
  },
  "baseline_warm": true
}
```

`baseline_warm: false` means fewer than 10 days of data are collected — deviations are less reliable.

## User Memory / Preferences

```json
{
  "preferences": {
    "work_hours": [9, 17],
    "max_daily_hours": 8.0,
    "goals": ["improve sleep", "reduce stress"],
    "preferred_interventions": ["yoga", "meditation", "walking"]
  },
  "wellbeing": {
    "risk_level": "mild",
    "trend": "stable",
    "history": [
      { "timestamp": "2024-01-15", "risk_level": "mild" },
      { "timestamp": "2024-01-14", "risk_level": "moderate" }
    ]
  }
}
```

Trend values: `stable` / `improving` / `declining`

## Feedback Record

Records user decision on a suggested change:

```json
{
  "user_id": "user@example.com",
  "suggestion": "Evening Wind-Down at 21:00",
  "action": "accepted",
  "timestamp": "2024-01-15T14:32:00",
  "behavioral_state_summary": "mild sleep disruption"
}
```

`action` values: `accepted` / `rejected`

Last 10 feedback records are included in each pipeline call for personalization.

## Sensor Log (mobile, stored locally)

```kotlin
data class SensorLog(
  val id: Long,
  val type: String,     // "motion", "location", "wifi"
  val timestamp: Long,  // Unix milliseconds
  val content: String   // JSON payload
)
```

Motion payload example:
```json
{ "activity": "Walking", "confidence": 0.92, "acceleration_ms2": 2.3 }
```

Location payload example:
```json
{ "lat": 30.266, "lon": -97.742, "accuracy_m": 15.0, "cluster": "home" }
```
